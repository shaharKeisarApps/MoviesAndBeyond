# v0.0.2 Release — Agent Team Orchestration Prompt

## How to Use

Copy the prompt below into Claude Code to launch the v0.0.2 release team. The session running this prompt becomes the **Team Lead** automatically.

### Prerequisites
- `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` is already set in `.claude/settings.json` ✅
- All agents are defined in `.claude/agents/` ✅
- Release spec exists at `.claude/specs/features/v0.0.2-ui-landscape-release.md` ✅
- `device-validate` skill exists at `.claude/skills/device-validate/SKILL.md` ✅
- Connected Android device or emulator running (for runtime validation) ✅

## The Prompt

```
You are the Release Lead for MoviesAndBeyond v0.0.2. This is a CRITICAL release focused on UI polish and landscape support. Your job is to orchestrate a 5-agent team to deliver this release with high quality, full test coverage, and ON-DEVICE validation.

Read these files first:
- `.claude/specs/features/v0.0.2-ui-landscape-release.md` (release requirements)
- `.claude/specs/tasks/task-completion-gates.md` (quality gates)
- `.claude/skills/device-validate/SKILL.md` (device validation protocol — CRITICAL for this release)

IMPORTANT: This release is UI-focused. Code compilation alone is NOT sufficient. Every fix must be validated ON DEVICE using the device-validate skill's 4-phase protocol. The device-validate skill uses adb + uiautomator to verify insets, layout spacing, accessibility, and product flows on a real device/emulator.

Then create an agent team with 5 teammates:

1. **"ui-fixer"** — Edge-to-edge & inset fix specialist
   Prompt: "You are fixing edge-to-edge and WindowInsets issues across all screens in MoviesAndBeyond. Read `.claude/agents/ui-landscape-specialist.md` and `.claude/specs/features/v0.0.2-ui-landscape-release.md` for full context.

   CRITICAL CONTEXT — Read `.claude/skills/device-validate/SKILL.md` section 'Edge-to-Edge Inset Chain' and 'Known Gotcha: TopAppBar Double Inset'. The current inset architecture is:
   1. MainActivity: enableEdgeToEdge() + setDecorFitsSystemWindows(false)
   2. MoviesAndBeyondApp: HazeScaffold with contentWindowInsets = WindowInsets.safeDrawing
   3. MoviesAndBeyondNav3: Modifier.padding(top = paddingValues.calculateTopPadding()) on NavDisplay
   4. Inner Scaffolds: contentWindowInsets = WindowInsets(0, 0, 0, 0) — this is INTENTIONAL to avoid double-consuming
   5. TopAppBars must set windowInsets = WindowInsets(0, 0, 0, 0) to avoid double status bar padding

   The REAL bug is NOT the zeroed WindowInsets on inner scaffolds — that's by design. The bug is that bottom navigation bar insets are consumed by the outer HazeScaffold but the floating bottom bar OVERLAYS content (doesn't push it up). So the last items in scrollable lists get hidden behind the floating nav bar.

   Your tasks:
   - Add bottom content padding to all scrollable lists equal to the floating nav bar height (~96dp = 80dp bar + 8dp*2 padding) so last items are visible
   - Fix Details screen: ensure the bottom of the scrollable content has enough padding to clear the floating nav bar
   - Fix Credits/Cast screen: add bottom padding to LazyColumn so last cast members aren't hidden behind the floating nav bar
   - Do NOT change contentWindowInsets = WindowInsets(0, 0, 0, 0) on inner scaffolds — the outer HazeScaffold handles system insets
   - Do NOT add navigationBarsPadding() to inner content — this would double-consume since HazeScaffold already handles it
   - Verify TopAppBars have windowInsets = WindowInsets(0, 0, 0, 0) to avoid double status bar padding

   Files to fix: feature/movies/FeedScreen.kt, feature/tv/FeedScreen.kt, feature/search/SearchScreen.kt, feature/details/DetailsScreen.kt, feature/details/CreditsScreen.kt, feature/details/content/MediaDetailsContent.kt, feature/details/content/PersonDetailsContent.kt, feature/auth/AuthScreen.kt, feature/you/YouScreen.kt, feature/you/library_items/LibraryItemsScreen.kt

   After fixing, run: ./gradlew :feature:movies:compileDebugKotlin :feature:tv:compileDebugKotlin :feature:search:compileDebugKotlin :feature:details:compileDebugKotlin :feature:auth:compileDebugKotlin :feature:you:compileDebugKotlin
   Report what you changed and any compilation issues."

2. **"landscape-architect"** — Landscape & adaptive layout specialist
   Prompt: "You are adding landscape mode support to MoviesAndBeyond. Read `.claude/agents/ui-landscape-specialist.md` and `.claude/specs/features/v0.0.2-ui-landscape-release.md` for full context.

   Also read `.claude/skills/device-validate/SKILL.md` to understand the current inset chain architecture — do NOT break it when adding landscape support.

   Your tasks:
   - Remove `android:screenOrientation='portrait'` from app/src/main/AndroidManifest.xml
   - Add `material3-window-size-class` dependency to the version catalog and core:ui module
   - Calculate WindowSizeClass in MainActivity and pass it through the app via CompositionLocal
   - Create `core/ui/src/main/java/.../core/ui/AdaptiveLayout.kt` with shared responsive utilities
   - Adapt the FloatingNavigationBar for landscape (consider NavigationRail for Expanded width)
   - Update HazeScaffold and MoviesAndBeyondApp to handle landscape insets (displayCutout on sides)
   - Make feed screens (Movies, TV) use adaptive grid columns based on WindowSizeClass
   - Make SearchScreen grid adaptive
   - Make Details screen use two-pane layout in landscape (backdrop left, content right)
   - Make Credits screen use wider grid in landscape
   - Ensure Auth, You, LibraryItems, and Onboarding screens work in landscape
   After implementing, run: ./gradlew compileDebugKotlin
   Report the full list of files created/modified and any issues."

3. **"quality-validator"** — Build & quality gate enforcement
   Prompt: "You are the build quality validator for MoviesAndBeyond v0.0.2. Wait for the ui-fixer and landscape-architect teammates to report completion, then run the full quality gate sequence:
   Gate 1 — Compilation: ./gradlew compileDebugKotlin
   Gate 2 — Code quality: ./gradlew spotlessApply && ./gradlew spotlessCheck detekt lint
   Gate 3 — Unit tests: ./gradlew test
   Gate 5 — Integration: ./gradlew clean assembleDebug test check
   If any gate fails, message the responsible teammate with the exact errors so they can fix them. Keep re-validating until all gates pass.
   Also review for:
   - Consistent WindowInsets usage (inner scaffolds should keep WindowInsets(0,0,0,0) — outer HazeScaffold handles system insets)
   - WindowSizeClass properly threaded through all screens
   - No hardcoded column counts in grids (use Adaptive or WindowSizeClass-based)
   - Proper landscape handling in all feature modules
   Once all gates pass, message the device-validator teammate that the APK is ready for on-device testing.
   Report final gate status for each gate."

4. **"device-validator"** — ON-DEVICE runtime validation (CRITICAL)
   Prompt: "You are the device validation specialist for MoviesAndBeyond v0.0.2. This is the MOST CRITICAL teammate for this release because v0.0.2 is a UI-focused release — code compilation alone is NOT sufficient proof of correctness.

   Read `.claude/skills/device-validate/SKILL.md` thoroughly — it is your complete protocol.

   Wait for the quality-validator teammate to confirm all build gates pass and the debug APK is ready. Then execute the FULL 4-phase device-validate protocol:

   **Phase 1: Code Analysis** (can start immediately, before APK)
   Run all 4 sub-analyses in parallel:
   - 1a: Material Design 3 compliance — M3 components, color tokens, typography, spacing
   - 1b: Compose patterns — Modifier params, state hoisting, recomposition stability, lifecycle
   - 1c: Accessibility — contentDescription, touch targets, labels
   - 1d: Inset architecture — verify the full inset chain per SKILL.md section 'Edge-to-Edge Inset Chain'

   **Phase 2: Product Acceptance Testing** (requires device)
   Install APK: adb install -r app/build/outputs/apk/debug/app-debug.apk
   Execute ALL 5 product scenarios from SKILL.md:
   - P1: Tab Navigation (all 4 tabs render, bottom nav visible)
   - P2: Details Flow (movie tap → details → cast → credits → back)
   - P3: Library Flow (You → login if needed → favorites/watchlist)
   - P4: Search Flow (search → type → results → tap → details → back with query preserved)
   - P5: Settings Dialog (You → settings → toggles → dismiss)

   Run each scenario in BOTH orientations:
   - Portrait: adb shell settings put system accelerometer_rotation 0 && adb shell settings put system user_rotation 0
   - Landscape: adb shell settings put system accelerometer_rotation 0 && adb shell settings put system user_rotation 1
   Take screenshots after each navigation step: adb shell screencap -p /sdcard/screen.png && adb pull /sdcard/screen.png

   **Phase 3: Runtime Technical Validation** (requires device)
   Execute ALL 6 validations from SKILL.md:
   - V1: All screens render (navigate all tabs, screenshot each)
   - V2: Keyboard spacing on Search (gap <= 24dp between field and keyboard)
   - V3: Layout spacing (no unexplained empty views >500px wide AND >300px tall)
   - V4: Accessibility scan (all clickable nodes have content-desc or text)
   - V5: Touch target size (all clickable bounds >= 48x48dp)
   - V6: Inset verification (TopAppBar positions match expected per-screen values)

   Run V1-V6 in BOTH portrait AND landscape orientations.

   **Phase 4: Cross-Reference Report**
   Generate the full report with:
   - Inset Health Check table (per-screen TopAppBar position vs expected)
   - Edge-to-Edge Verification table
   - Landscape Adaptation table (new for v0.0.2):
     | Screen | Portrait Status | Landscape Status | Adaptive Grid | Nav Type |
   - All FAIL results must be messaged to the responsible teammate (ui-fixer or landscape-architect) with:
     a) The exact screen and orientation
     b) Screenshot evidence
     c) Expected vs actual behavior
     d) The specific SKILL.md threshold that was violated

   Keep re-validating after fixes until ALL phases produce PASS across both orientations.
   Final report must show 100% PASS rate before v0.0.2 can ship."

5. **"code-reviewer"** — Architecture & pattern compliance
   Prompt: "You are the code reviewer for MoviesAndBeyond v0.0.2. Read `.claude/agents/code-reviewer.md` for your full review protocol. Also read `.claude/skills/device-validate/SKILL.md` to understand the inset architecture you're reviewing.

   Wait for BOTH the quality-validator to confirm all build gates pass AND the device-validator to confirm Phase 1 (code analysis) is clean. Then do a comprehensive code review of all changes. Focus on:
   - Material 3 adaptive layout best practices
   - Inset chain correctness (per device-validate SKILL.md 'Edge-to-Edge Inset Chain'):
     * Outer HazeScaffold: contentWindowInsets = WindowInsets.safeDrawing ✓
     * NavDisplay: top padding from paddingValues ✓
     * Inner Scaffolds: contentWindowInsets = WindowInsets(0, 0, 0, 0) ✓
     * Inner TopAppBars: windowInsets = WindowInsets(0, 0, 0, 0) ✓
     * Details screen exception: backdrop edge-to-edge ✓
   - WindowSizeClass usage consistency across all screens
   - State preservation during orientation changes (WhileSubscribedOrRetained, rememberSaveable)
   - Compose recomposition efficiency (no unnecessary recompositions from orientation change)
   - Navigation patterns in landscape (bottom bar vs rail transition)
   - Accessibility in both orientations (from device-validate Phase 1c findings)
   Report findings as: Blockers (must fix), Suggestions (should fix), Nitpicks (nice to have).
   If there are Blockers, message the responsible teammate to fix them."

## Coordination Rules

As the Team Lead, follow these rules:

1. **Launch order**:
   - Wave 1 (parallel): Spawn ui-fixer + landscape-architect (implementation)
   - Wave 1 (parallel): Spawn device-validator (starts Phase 1 code analysis immediately — no APK needed)
   - Wave 2 (after implementation): quality-validator runs build gates
   - Wave 2 (after gates pass): device-validator continues to Phases 2-3-4 (needs APK)
   - Wave 3 (after device Phase 1 + build gates): code-reviewer starts

2. **Conflict resolution**: If ui-fixer and landscape-architect both need to modify the same file, have landscape-architect defer to ui-fixer's inset fixes first, then layer landscape changes on top.

3. **File ownership**:
   - ui-fixer OWNS: Bottom padding for floating nav bar, inset fixes, edge-to-edge corrections
   - landscape-architect OWNS: WindowSizeClass, adaptive grids, orientation removal, new layout utilities
   - device-validator OWNS: Runtime truth — its Phase 3/4 results OVERRIDE code-level assumptions
   - Shared files (FeedScreen, DetailsScreen, etc.): ui-fixer goes first, landscape-architect follows

4. **Device-validate authority**: The device-validator has VETO power over the release. Even if all build gates pass and code review is clean, if device-validate Phase 3/4 shows FAIL results, the release is blocked. The device-validator's screenshots and measurements are the ground truth for UI correctness.

5. **Gate enforcement**: No PR is created until:
   - quality-validator confirms ALL build gates pass
   - code-reviewer confirms zero Blockers
   - device-validator confirms 100% PASS on all 4 phases in BOTH orientations

6. **Re-iteration loop**: If any validator finds issues:
   - device-validator messages responsible teammate with screenshot evidence
   - Teammate fixes the issue
   - quality-validator re-runs build gates (fast — only if code changed)
   - device-validator re-runs the FAILED phases only (not full re-validation)
   - Maximum 3 iterations per issue before escalating to lead

7. **Final synthesis**: Once ALL 5 teammates report success, synthesize a release summary with:
   - All files modified
   - All UI issues fixed (with before/after evidence from device-validator screenshots)
   - Landscape support status per screen (from device-validate Phase 4 Landscape table)
   - Inset Health Check results (from device-validate Phase 4)
   - Build quality gate results
   - Code review verdict
   - Device validation final report (100% PASS confirmation)
   - Any remaining items for v0.0.3

Wait for all teammates to complete before declaring v0.0.2 ready.
```

## Post v0.0.2: KMP/iOS Planning Prompt

After v0.0.2 ships, use this follow-up prompt to begin KMP planning:

```
Now that v0.0.2 is shipped with UI fixes and landscape support, let's plan the KMP/iOS migration for v0.1.0.

Create an agent team with 3 teammates:

1. **"kmp-architect"** — Plan the module-by-module KMP migration strategy
   Analyze the current module structure and propose:
   - Which modules can be shared (core:model, core:network, data)
   - Which remain platform-specific (core:ui, core:local partially, all features)
   - expect/actual boundaries needed
   - Dependency replacements (Hilt → Koin/Kodein or Metro, Room → SQLDelight, etc.)
   - Build system changes (convention plugins, Kotlin targets)
   - Migration order (core first, then data, then features)

2. **"ios-specialist"** — Plan the iOS-specific implementation
   Design the iOS app architecture:
   - SwiftUI vs Compose Multiplatform for UI
   - iOS navigation patterns
   - Platform-specific integrations (Keychain, UserDefaults, URLSession)
   - Xcode project structure
   - CocoaPods vs SPM for shared module consumption

3. **"migration-validator"** — Risk assessment & feasibility
   Evaluate:
   - Breaking change impact on existing Android app
   - CI/CD changes needed
   - Testing strategy for shared code
   - Timeline estimate per module
   - Risk matrix (high/medium/low per module)

Produce a comprehensive KMP Migration Plan document at `.claude/specs/features/kmp-ios-migration-plan.md`.
```
